# 📡 HFSonar 设计与功能说明

> 用一份文档把 HFSonar **能做什么**、**怎么做的**、**为什么这么做** 讲清楚。
> 看完之后,你应该能对任何问题做出判断:这个改动该不该加、这个 bug 出在哪一层、这个扩展该往哪个文件下手。

---

## 一、它是什么(一句话定位)

HFSonar 是一个**给研究者用的"自动 networking 助手"**——它运行在本地,每隔一段时间(默认 1 小时):

> 把 HuggingFace 上当下值得关注的内容(新模型、新论文、热门趋势、关注 org 的新动作)抓下来,**让 Claude 帮你判断"哪几个项目背后的人是你想认识的"**,然后**让 Claude 用 WebFetch / WebSearch 帮你扒出这些人的公开联系方式**(邮箱、Twitter、GitHub、个人主页),**最后帮你写一封个性化的 cold-outreach 信**。

输出最终落到本地磁盘,**不会自动发出去**。你的角色是看一眼草稿、决定 send / kill / edit、自己按"发送键"。

它不是"自动发推"工具,是"**自动找该认识的人 + 自动写 connect 信**"工具。社交的判断力(谁值得认识、要不要发、用什么口吻调整)留给你;最耗时的发现 + 起草留给 Claude。

## 二、它能做什么(功能清单)

### 1. 🛰️ 多信源实时监控

每个 cycle 同时从 4 个 HuggingFace 信源抓取候选:

| 信源 | 数据来源 | 默认抓多少 | 用途 |
| --- | --- | --- | --- |
| **Trending Models** | `huggingface_hub.list_models(sort="trendingScore")` | 20 条 | 当下最火的模型 |
| **New Models** | `list_models(sort="createdAt")` | 30 条 | 最新创建的模型(捕捉刚发布的 release) |
| **Daily Papers** | `huggingface.co/api/daily_papers` | 15 条 | HF 编辑团队每日精选论文 |
| **Watchlist Orgs** | `list_models(author=org)` 遍历配置的 org | 每个 org 3 条 | 盯紧 Meta / Mistral / Qwen / DeepSeek 等 10 家实验室 |

每个信源独立异常隔离——某个挂了,其他三个继续。

### 2. 🗂️ 跨运行去重(不重复打扰同一个人)

每个被处理过的内容(以 `source::event_id` 为唯一键)都写进 `state/ledger.jsonl`。

下一次 cycle 起来时,所有候选先过一遍 ledger,**已经为它写过 outreach 的直接丢掉**——不会重复给同一个研究者写信。

### 3. 🧠 Curator:筛选"值得认识的人"

抓到的几十条候选先过 `min_likes=5` 粗过滤(watchlist 和 papers 豁免),剩下的扔给 Claude 当 **Curator** 角色:

```
你是 HFSonar 的 Curator。HFSonar 在帮一个研究者建立人脉网络。
从这批候选里挑出 K 条,这些项目背后的人是真正值得发 connect 信的。
对每一条,给一句话的"angle"——outreach 的钩子。
```

Curator 在 prompt 里被明确教导**拒绝**:

- 匿名 re-uploads / 量化分叉(GGUF/GPTQ/AWQ 镜像)→ 上传者不是原作者
- 大厂团队发布且无具名作者(找不到具体人发邮件)→ 联系方式只会是 `support@company.com`
- 空仓库 / 占位 demo / merge 类合并模型

**偏好**:

- 有 corresponding author 的 Daily Paper
- 小团队或独立研究者的精心之作(model card 写得清楚、有真 eval、有有趣的设计选择)
- 解决了不寻常问题、提出新架构的工作——这种作者最值得 1-on-1 聊

输出是结构化 JSON,主程序解析后只对被选中的 K 条往下走。

### 4. 🕵️ Researcher:自动扒联系方式

这是 v2 新加的核心阶段。Curator 选出的每条,都用一个独立 prompt 调用 Claude 当 **Researcher** 角色,**并允许它使用 `WebFetch` 和 `WebSearch` 工具**:

```
给定一个 HF artifact (model 或 paper),找出背后的"人"和公开联系方式。
- HF 用户/组织主页
- arXiv abstract 页(论文 corresponding author)
- GitHub 主页
- 个人/实验室网站
- 必要时 Google search

铁律:
- 永远不编造邮箱(不要猜 firstname@university.edu 这种 pattern)
- 优先公开 handle (Twitter/GitHub) 而不是被扒出来的邮箱
- 每条联系方式都要在 notes 里写清楚来源
```

返回结构化 JSON:

```json
{
  "primary_author": {
    "name": "Alice Researcher",
    "role": "corresponding",
    "affiliation": "MIT CSAIL",
    "email": "alice@csail.mit.edu",
    "twitter": "@alice_ml",
    "github": "alice",
    "website": "alice.dev",
    "huggingface": "https://huggingface.co/alice",
    "confidence": "high"
  },
  "coauthors": [...],
  "notes": "Email from arXiv corresponding-author line; twitter from HF profile; github cross-checked."
}
```

`confidence` 字段(`high | medium | low | name_only`)让你一眼看出哪些联系方式可信、哪些只是"找到名字了但联系方式靠猜"。

### 5. ✍️ Drafter:写 channel-aware 的 outreach 信

第三阶段调用 Claude 当 **Outreach Drafter**,把 Curator 的 angle、原始 event 信息、Researcher 的 author JSON 全部喂进去,产出最终草稿:

**Channel 自适应**:

- 找到邮箱 → 写邮件(带 Subject 行)
- 只有 Twitter → 写 DM(开头 ≤ 280 字符,可直接公开 reply)
- 只有 GitHub → 写一条礼貌的 issue / discussion 评论
- 多个渠道都有 → 默认邮件 + 在末尾列出"Alternative channels"

**Voice**(`.claude/skills/guides/cold-outreach-tone.md` 里写死):

- ❌ 不写 "Hope this email finds you well" / "groundbreaking" / "I'm a huge fan"
- ❌ 不要求 30 分钟会议(low-friction ask:"if you ever have a minute" / "no pressure to reply")
- ❌ 不假装认识对方,不替你编个人背景(用 `<your name>` `<your background>` 占位符,你填)
- ✅ 第一句必须是"I read your XXX, the way you handled YYY caught my attention"——用具体细节证明你真的看了
- ✅ 体长 3-5 句话,邮件正文 < 100 字
- ✅ 一个具体的观察 OR 一个具体的问题——不是三个

### 6. 📚 项目专属"知识"自动注入

`.claude/skills/guides/` 目录里有三份 markdown:

| 文件 | 内容 |
| --- | --- |
| `cold-outreach-tone.md` | Cold-outreach 的 voice:不写"groundbreaking"、要带 specific hook、low-friction ask |
| `outreach-format.md` | 输出格式约束:`**Channel:**` / `**To:**` / `**Subject:**` 必填行,Alternative channels 段格式 |
| `hf-context.md` | HF 生态背景 + **去哪找 author 联系方式**(arXiv PDF / HF profile / GitHub / Google Scholar 各自的可靠度) |

**Claude Code 在这个仓库里跑的时候会自动加载这些文件**——不需要在 prompt 里手动 include。这是 Claude Code 的内建机制。

### 7. 🔁 完全可复现的运行记录

每次 cycle 在 `runs/` 下生成一个独立目录(以 UTC 微秒时间戳命名,不会冲突):

```
runs/20260509T023827_714106Z/
├── prompts/
│   ├── 01_curate.prompt.md          # 发给 Claude 的 Curator 完整 prompt
│   ├── 02_research_01.prompt.md     # 发给 Claude 的第 1 条 Researcher prompt
│   ├── 02_research_02.prompt.md
│   ├── 02_research_03.prompt.md
│   ├── 03_outreach_01.prompt.md     # 发给 Claude 的第 1 篇 Outreach prompt
│   ├── 03_outreach_02.prompt.md
│   └── 03_outreach_03.prompt.md
├── outreach/
│   ├── 01__daily_papers__2605.06627.md
│   ├── 02__watchlist__Qwen__Qwen3-Next-7B.md
│   └── 03__trending_models__author__model.md
└── run_manifest.json                # 候选列表、curator 输出、research 摘要、最终路径
```

每篇 outreach 草稿都带 JSON frontmatter,包含:event 信息 + 完整 author research 结果 + 来源说明 + 生成时间。

如果某条草稿写得不好,可以:

```bash
# 直接重跑同一个 prompt
claude -p @runs/<ts>/prompts/03_outreach_03.prompt.md

# 改 prompt 模板再重跑
```

### 8. 🧪 0 token 的 dev/CI 模式

`--fake-llm` 让整个 3 阶段 pipeline 用 `FakeOperator` 替代真 Claude。fetch、dedup、curate prompt 渲染、author research 模拟、outreach prompt 渲染、frontmatter 拼接、ledger 写入全跑,只是 LLM 被替换成读 prompt → 返回固定 JSON / markdown。

13 个 pytest 测试 < 0.2 秒全过。

### 9. 🚀 三种运行方式

```bash
python main.py poll              # 跑一次,退出
python main.py poll --fake-llm   # 跑一次,不花钱
python main.py loop --interval 3600   # 一直跑,每小时一轮(SIGINT 优雅停止)
python main.py list              # 列出过去所有 cycle
```

## 三、设计哲学

### 原则 1:**Claude 是大脑,Python 是骨架**

Python 主程序**不做任何"判断"工作**——它只做:抓数据、过滤已发的、组装 prompt、调 Claude、写文件。

所有需要"是不是值得发"、"这个人值不值得认识"、"邮箱去哪找"、"信怎么写"的判断,全丢给 Claude。

### 原则 2:**永远不编造联系方式**

这是**最硬的红线**。Researcher prompt 里写死:

- 找不到邮箱就空着,不要猜 `firstname.lastname@university.edu` 这种 pattern
- 优先公开 handle (Twitter / GitHub) 而不是 PDF 里扒的邮箱
- 每条联系方式都要在 `notes` 里说明来源

为什么这条这么重要:**冷邮件给一个错的地址 = 毫无意义,但冷邮件给一个不愿被打扰的真实地址 = 骚扰**。两者都要避免。

### 原则 3:**Subprocess CLI,不要 SDK**

```python
subprocess.run(["claude", "-p", "@/path/to/prompt.md", "--output-format", "json"])
```

不引入 anthropic SDK,不管理 API key,直接复用你 Claude Code 的登录态和计费。

第二阶段(Researcher)需要 web 工具,所以加了 `--allowedTools WebFetch,WebSearch` ——只白名单**只读**工具,不开 shell、不开 Edit。

### 原则 4:**Run dir 是真相之源**

发给 Claude 的每个字、Claude 返回的每个字,都落盘。调试 bug 永远不靠"复述"。

### 原则 5:**默认是 dry-run,绝不自动发**

v1 故意不接任何邮件 / Twitter / LinkedIn 发送 API。原因:

- **自动发冷邮件 = 垃圾邮件**——技术上能做,道德上不该做
- HFSonar 解决的是"发现 + 起草",不是"发送"
- 你按下"发送键"的那一秒,等于你声明"我读过、我同意"——这个责任不能让 LLM 担

如果某天你想加个"半自动"模式(比如"发到我自己的草稿箱,我从那里 review"),那是新增一个 `Sender` 模块的事。但默认状态永远是 dry-run。

### 原则 6:**简单战胜花哨**

- ledger 用 JSONL,不用 SQLite
- 调度用 `time.sleep(interval)`,不用 cron / systemd-timer
- 配置用 `tomllib`(Python 3.11+ 标准库),不用 pydantic
- 没有 web UI,只有 `list` 子命令
- 没有 plugin 系统,只有一个 `Source` 抽象类 + 4 个具体实现

## 五、模块详解

### `src/events.py` — 统一事件结构

`Event` dataclass 把 4 个信源的数据归一化。`dedup_key` = `{source}::{event_id}`,用于跨运行去重。

### `src/ledger.py` — 跨运行去重

启动时把 JSONL 读进 `set`,运行期 `has`/`filter` 都是 O(1)。`record` 既写文件又更新 set,**幂等**。

### `src/sources/*.py` — 4 个信源适配器

每个文件实现 `Source` 抽象类的 `fetch(limit) -> list[Event]`。失败返回 `[]` 而不是 raise。`_hf_common.py` 集中放 `ModelInfo → Event` 的转换,以后 HF SDK 改 schema 只改这一个文件。

### `src/operator.py` — Claude CLI 包装层

```python
class Operator(Protocol):
    def run(self, prompt_path: Path, *, label: str = "",
            tools: list[str] | None = None) -> OperatorResult: ...
```

`tools` 参数是 v2 加的——orchestrator 调 research 阶段时传 `tools=["WebFetch", "WebSearch"]`,其他两个阶段不传(纯文本 in/out)。

具体实现两个:

- **`ClaudeOperator`**:`subprocess.run(["claude", "-p", "@file", "--output-format", "json"])`,如果 `tools` 非空就加 `--allowedTools tools`
- **`FakeOperator`**:读 prompt,看里面有 `[ROLE: CURATOR]` / `[ROLE: RESEARCHER]` / `[ROLE: OUTREACH_DRAFTER]` 中的哪个标记,返回对应的固定假输出

### `src/orchestrator.py` — 编排核心

`run_cycle` 是入口,做这些事:

```
1. _utc_run_id() → 生成微秒级时间戳作为 run id
2. fetch_all()    → 从 4 个信源抓数据
3. filter_candidates() → ledger 去重 + min_likes 阈值过滤
4. 组装 curate prompt → 调 operator → 解析 JSON 结果
5. 对每条被 curator 选中的 event:
   a) 组装 research prompt → 调 operator(带 WebFetch/WebSearch)→ 解析 author JSON
   b) 组装 outreach prompt(把 angle + event + author JSON 全塞进去)→ 调 operator → 拿 markdown body
   c) 写 outreach 文件(JSON frontmatter + body)
   d) ledger 记录这条已处理
6. 写 run_manifest.json
```

`_parse_curator_output` 和 `_parse_research_output` 都基于 `_extract_json_object`——宽容的 JSON 解析器,能处理裸 JSON / fenced JSON / prose 内嵌 JSON 三种形式。

### `src/prompts/` — Prompt 模板

不用 Jinja2 不用任何模板引擎,**就是 `text.replace("{{KEY}}", value)`**。

| 文件 | 角色 | 占位符 |
| --- | --- | --- |
| `01_curate.md` | `[ROLE: CURATOR]` | `CYCLE_TIMESTAMP`, `NUM_CANDIDATES`, `TOP_K`, `CANDIDATES_BLOCK` |
| `02_research_author.md` | `[ROLE: RESEARCHER]` | `SOURCE`, `TITLE`, `URL`, `AUTHOR`, `CREATED_AT`, `SUMMARY` |
| `03_draft_outreach.md` | `[ROLE: OUTREACH_DRAFTER]` | `SOURCE`, `TITLE`, `URL`, `LIKES`, `CREATED_AT`, `TAGS`, `SUMMARY`, `ANGLE`, `AUTHOR_JSON` |

改 prompt 不需要改 Python。

### `main.py` — CLI 入口

`argparse`,3 个子命令(`poll` / `loop` / `list`)。loop 模式安装 SIGINT/SIGTERM handler,sleep 时按 1 秒一片切,确保 Ctrl+C 响应。

## 六、一次完整 cycle 走读

假设你在终端跑 `python main.py poll`,从那一刻开始:

```
T+0.00s   读 config.toml
T+0.01s   ClaudeOperator 实例化(检查 PATH 上有 claude)
T+0.02s   Ledger 加载 state/ledger.jsonl,假设里面有 1500 条历史 key
T+0.03s   生成 run_id = "20260509T023827_714106Z"

[fetch 阶段]
T+0.20s   trending_models 拉到 20 条
T+0.50s   new_models 拉到 30 条
T+1.20s   daily_papers 拉到 15 条
T+3.50s   watchlist 串行扫 10 个 org,共拿到 27 条
          总计 fetched = 92 条 events

[filter 阶段]
T+3.51s   按 dedup_key 去重 → 87 条
T+3.52s   去掉 ledger 已有的 → 41 条
T+3.53s   去掉 likes < 5 的 trending/new → 32 条候选

[curate 阶段]
T+3.55s   渲染 01_curate.prompt.md(把 32 条候选信息嵌入模板)
T+3.56s   subprocess: claude -p @01_curate.prompt.md --output-format json
T+12.4s   Claude 返回 JSON,parse 出 chosen = 3 条 + 每条的 angle

[research × 3 阶段]
T+12.5s   渲染 02_research_01.prompt.md(填入第 1 条的 event 信息)
T+12.5s   subprocess: claude -p @02_research_01.prompt.md \
                       --output-format json --allowedTools WebFetch,WebSearch
T+25.0s   Claude 用 WebFetch 抓了 HF profile + arXiv abstract,
          parse 出 primary_author JSON
... × 2 次 ...

[outreach × 3 阶段]
T+50.0s   渲染 03_outreach_01.prompt.md(把 angle + event + author JSON 全塞进去)
T+50.1s   subprocess: claude -p @03_outreach_01.prompt.md --output-format json
T+58.0s   Claude 返回 markdown 草稿
T+58.1s   写 runs/.../outreach/01__source__id.md(JSON frontmatter + body)
T+58.1s   ledger.record(key) → state/ledger.jsonl 追加一行
... × 2 次 ...

[收尾]
T+90.0s   写 runs/.../run_manifest.json
T+90.0s   stdout 打印 "outreach drafts written: 3"
T+90.0s   进程退出
```

整个 cycle 约 60–120 秒,主要时间花在 7 次 Claude CLI 调用上(1 curate + 3 research + 3 outreach)。Research 阶段最慢——它要 WebFetch / WebSearch。

## 七、配置项

`config.toml`(可被 `config.local.toml` 覆盖,后者 gitignore):

| 配置 | 默认值 | 含义 |
| --- | --- | --- |
| `[poll] trending_limit` | 20 | trending 信源每 cycle 抓多少 |
| `[poll] new_models_limit` | 30 | new 信源每 cycle 抓多少 |
| `[poll] daily_papers_limit` | 15 | papers 信源每 cycle 抓多少 |
| `[poll] watchlist_limit` | 3 | watchlist **每个 org** 抓多少 |
| `[curation] top_k` | 5 | curator 最多保留几条 |
| `[curation] min_likes` | 5 | trending/new 的赞数阈值(papers/watchlist 豁免) |
| `[watchlist] orgs` | 10 家 | 关注的 HF 组织 |
| `[claude] binary` | `"claude"` | CLI 名字 |
| `[claude] timeout` | 180 | 单次 CLI 调用超时(秒) |
| `[claude] model` | `""` | 模型覆盖,空 = 用 Claude Code 的默认 |

## 八、扩展指南

### 加一个新信源(比如 HF Spaces)

1. 在 `src/sources/` 加 `spaces.py`,实现 `Source` 接口
2. 在 `events.py` 加 `SOURCE_SPACES = "spaces"`
3. 在 `orchestrator.build_sources()` 加一行
4. 在 `config.toml` 的 `[poll]` 加 `spaces_limit = ...`
5. 不需要改 curator/researcher/drafter/ledger/operator——它们都看的是统一的 `Event` 结构

### 加一个 Sender(比如发到自己的 Gmail Drafts)

1. 创建 `src/senders/gmail_drafts.py`,接 Gmail API,把 `outreach.md` 拆成 to/subject/body 写到 Drafts 文件夹(**不是 Sent**——人在那一步才决定发不发)
2. 在 `orchestrator.run_cycle()` 末尾,在 `write_outreach_file()` 之后加一句条件调用
3. 在 CLI 加 `--draft-to gmail` 标志

### 换掉 Claude

1. 在 `src/operator.py` 加一个 `OpenAIOperator` 类,实现 `Operator` Protocol
2. CLI 加 `--backbone openai` 标志

Curator / Researcher / Outreach prompts 都是纯文本,任何 LLM 都吃。注意 OpenAI 的工具调用方式和 Claude 的 `--allowedTools` 不一样,research 阶段需要适配。

## 九、测试策略

13 个 pytest 测试,全部不需要网络、不需要 token:

| 文件 | 测试内容 |
| --- | --- |
| `test_ledger.py` | 写读、幂等、`filter_unseen` |
| `test_events.py` | dedup_key、dict round-trip、未知字段容忍 |
| `test_curator_parse.py` | curator 各种 JSON 形式 + research JSON 默认值填充 |
| `test_orchestrator_fake.py` | FakeOperator + stub source 的 3 阶段 e2e + 第二轮全去重 |

故意**不测真 Claude 调用**——那部分依赖外部进程,容易 flaky 也烧 token。靠 `subprocess.run` 的 stdlib 稳定性兜底,加上一次手动 `claude -p @file` smoke 验证 JSON shape 即可。

## 十、故意不做的事

- ❌ **不接 SMTP / Twitter / LinkedIn 发送 API** ── 自动发冷邮件 = 垃圾邮件,这条永远不做
- ❌ **没有 Web UI** ── `python main.py list` + 文件浏览器够用
- ❌ **没有 PDF 解析** ── Researcher 抓 arXiv abstract 页但不解 PDF 第一页;有需要再加 `pypdf`
- ❌ **没有多语言 outreach** ── Drafter prompt 写死英文(本身可以国际化,把 `cold-outreach-tone.md` 加一份中文版即可)
- ❌ **没有 CRM 集成** ── 没有 "Alice 回复了吗" 的追踪;接 Notion / Airtable 可以,但不是 v1
- ❌ **不替你按发送键** ── 这是设计原则,不是缺失功能

## 十一、一句话总结

> HFSonar 是个"用 Python 把 Claude Code 抱在一个 polling loop 里" 的最小可用 networking agent。
> 它把"刷 HuggingFace + 找出值得认识的研究者 + 扒他们的联系方式 + 写 connect 信" 这几件每天耗你半小时的活儿,自动化成 3-5 份磁盘上的草稿。
> 它故意不替你按"发送键"——因为按不按那个键,本来就该是你来决定的。
