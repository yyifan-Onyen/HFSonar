# 📡 HFSonar 设计与功能说明

> 用一份文档把 HFSonar **能做什么**、**怎么做的**、**为什么这么做** 讲清楚。
> 看完之后,你应该能对任何问题做出判断:这个改动该不该加、这个 bug 出在哪一层、这个扩展该往哪个文件下手。

---

## 一、它是什么

HFSonar 是一个 **运行在本地** 的 HuggingFace 监控代理(monitoring agent)。

它每隔一段时间(默认 1 小时)做这样一件事:

> 把 HuggingFace 上当下值得关注的内容(新模型、新论文、热门趋势、关注 org 的新动作)抓下来,让 Claude 帮你判断"哪几条值得发",再让 Claude 帮你把这几条**写成可以直接复制发布的 markdown 帖子**。

帖子最终落到本地磁盘,**不会自动发到任何社交平台**。

它的核心价值不是"自动发推",而是**节省你刷 HuggingFace 的时间**——把"发现值得发的东西 + 写第一版草稿"这两件最耗时间、最容易拖延的事自动化掉,你只负责最后一步:看一眼草稿,决定 ship 还是 kill。

## 二、它能做什么(功能清单)

### 1. 🛰️ 多信源实时监控

每个 cycle 同时从 4 个不同的 HuggingFace 信源抓取候选内容:

| 信源 | 数据来源 | 默认抓多少 | 用途 |
| --- | --- | --- | --- |
| **Trending Models** | `huggingface_hub.list_models(sort="trendingScore")` | 20 条 | 当下最火的模型 |
| **New Models** | `list_models(sort="createdAt")` | 30 条 | 最新创建的模型(捕捉刚发布的 release) |
| **Daily Papers** | `huggingface.co/api/daily_papers` | 15 条 | HF 编辑团队每日精选论文 |
| **Watchlist Orgs** | `list_models(author=org)` 遍历配置的 org | 每个 org 3 条 | 盯紧 Meta / Mistral / Qwen / DeepSeek 等 10 家实验室 |

每个信源失败都不会拖累其他信源——做了**独立异常隔离**,某一个信源挂掉,其他三个继续。

### 2. 🗂️ 跨运行去重

每发出过一次的内容(以 `source::event_id` 为唯一键),都会被写进 `state/ledger.jsonl` 这个**追加式日志**。

下一次 cycle 起来时,所有候选先过一遍 ledger,**已经发过的直接丢掉**。这意味着即使你跑 24 小时都不停,同一个模型/论文也不会被重复写成帖子。

为什么用 JSONL 而不是 SQLite?——

- 用 `tail -f state/ledger.jsonl` 就能看实时日志
- 用 `git diff` 就能看出去重历史
- 不需要 schema migration,加字段就是多写一个 key
- 单机几万条记录,set in-memory 完全够用

### 3. 🧠 Claude 智能筛选(Curator)

抓到的几十条候选**不会全部写**——`min_likes=5` 的粗过滤先去掉显然没人看的(watchlist 和 papers 豁免这条,因为它们的判断标准不同),剩下的扔给 Claude 当 **Curator** 角色:

```
你是 HFSonar 的 Curator。这是 N 条候选,从里面挑出最多 K 条值得发的。
对每一条,给一句话的"角度"(angle)——文案应该用什么钩子起笔。
```

Curator 的输出是**结构化 JSON**,主程序解析后只对被选中的 K 条调用 Writer——这样既限制了 token 消耗,也保证了发出的内容是 Claude 主动判断"值得"的,不是无脑机翻 HF API 数据。

Curator 在 prompt 里被明确教导:

- ❌ 拒绝:量化分叉(GGUF/GPTQ/AWQ 镜像,除非是某个大 release 的首发量化)、空仓库、占位 demo、merge 类合并模型
- ✅ 偏好:真正的新架构、新能力,大厂正式发布,Daily Papers 高赞,watchlist org 的任何动作

### 4. ✍️ Claude 自动撰写(Writer)

被 curator 选中的每条,都会用一个独立 prompt 调用 Claude 当 **Writer** 角色:

```
源信息:title / url / author / likes / created_at / tags / summary
Curator 给的角度:<一句话>
按照 .claude/skills/guides/ 里的 tone + format 规则,写一篇 markdown 帖子。
```

Writer 不直接看 Curator 的对话——是个全新的 session,只拿到精炼好的事实和角度。这样 Writer 不会被 Curator 的措辞污染,产出更干净。

输出是直接可发的 markdown,**主程序自动加 frontmatter**(source / event_id / url / likes / created_at / angle / generated_at)然后落到磁盘。

### 5. 📚 项目专属"知识"自动注入

`.claude/skills/guides/` 目录里有三份 markdown 文件:

| 文件 | 内容 |
| --- | --- |
| `ai-news-tone.md` | AI/ML 发布帖的写作 voice:不写"groundbreaking"、不堆 emoji、要带具体数字 |
| `post-formatting.md` | markdown 格式规约:模型 ID 用 backticks、论文标题用斜体、license 名字怎么写 |
| `hf-context.md` | HF 生态背景:trending 都是什么、什么是衍生新闻、watchlist orgs 是哪些 |

**Claude Code 在这个仓库里跑的时候会自动加载这些文件**——不需要在 prompt 里手动 include。这是 Claude Code 的内建机制,我们只是利用了它。

### 6. 🔁 完全可复现的运行记录

每次 cycle 在 `runs/` 下生成一个独立目录(以 UTC 微秒时间戳命名,不会冲突):

```
runs/20260508T222739_145839Z/
├── prompts/
│   ├── 01_curate.prompt.md          # 发给 Claude 的 Curator 完整 prompt
│   ├── 02_write_01.prompt.md        # 发给 Claude 的第 1 篇 Writer prompt
│   ├── 02_write_02.prompt.md
│   └── 02_write_03.prompt.md
├── posts/
│   ├── 01__trending_models__deepseek-ai__DeepSeek-V4-Pro.md
│   ├── 02__daily_papers__2605.06627.md
│   └── 03__watchlist__Qwen__Qwen3-Next-7B.md
└── run_manifest.json                # 候选列表、curator 输出、最终 post 路径
```

**所有发给 Claude 的 prompt 都原封不动落盘**。如果某条草稿写得不好,你可以:

```bash
# 直接重跑同一个 prompt,看是否稳定
claude -p @runs/<ts>/prompts/02_write_03.prompt.md

# 或者改一改 prompt 模板再重跑
```

这是设计核心之一:**run dir is the source of truth**——出问题就去 runs/ 翻日志,不用猜。

### 7. 🧪 0 token 的 dev/CI 模式

`--fake-llm` 标志让整个 pipeline 用一个**确定性 stub**(`FakeOperator`)替代真 Claude。fetch、dedup、prompt 渲染、文件落盘、ledger 写入全都跑,只是 LLM 被替换成读 prompt → 返回固定文本。

这意味着:

- 自动化测试 0 成本(11 个 pytest 全跑过 不到 0.1 秒)
- 改了主程序逻辑后,可以先 `--fake-llm` 验证整条管道不崩,再花钱真跑
- CI 可以定期跑 `--fake-llm` 确保 HF API schema 没变

### 8. 🚀 三种运行方式

```bash
python main.py poll              # 跑一次,退出
python main.py poll --fake-llm   # 跑一次,不花钱
python main.py loop --interval 3600   # 一直跑,每小时一轮(SIGINT 优雅停止)
python main.py list              # 列出过去所有 cycle
```

`loop` 模式会捕获 SIGINT/SIGTERM,**完成当前 cycle** 再停——不会留下半截写的 post。

## 三、设计哲学

整个项目的所有决策都围绕这 5 条原则:

### 原则 1:**Claude 是大脑,Python 是骨架**

Python 主程序**不做任何"判断"工作**——它只做:抓数据、过滤已发的、组装 prompt、调 Claude、写文件。

所有需要"是不是值得发"、"用什么角度"、"怎么遣词"的判断,都丢给 Claude。

这样的好处:
- 主程序代码极少(orchestrator 只有约 250 行 Python)
- 想调整"什么算值得发",只改 `01_curate.md` 这个文件,不用改 Python
- LLM 升级了,质量自动跟着升级,不用重新调代码

### 原则 2:**Subprocess CLI,不要 SDK**

Claude 的调用方式是:

```python
subprocess.run(["claude", "-p", "@/path/to/prompt.md", "--output-format", "json"])
```

**不引入 anthropic SDK,不管理 API key**。直接复用你 Claude Code 已有的登录态和计费。

为什么这么选:
- 0 配置——你能跑 Claude Code,就能跑 HFSonar
- 0 新依赖——Python 包只剩 `huggingface_hub` + `requests` + `pytest`
- 想换 backbone(Codex / 自部署模型 / GPT)?加一个 `Operator` 子类就行,Curator/Writer prompts 完全不动

### 原则 3:**Run dir 是真相之源**

发给 Claude 的每个字、Claude 返回的每个字,都落盘。

调试 bug 永远不靠"复述"——直接看 `runs/<ts>/`。

### 原则 4:**默认是 dry-run,不会自动发布**

v1 故意不接任何社交 API。原因:

- 接了 API 就有了**爆发风险**——bug 可能让你的账号刷屏
- 文案质量没稳定下来之前,自动发等于自残
- 写发布器是简单的工程问题,**判断什么值得发才是难的**——HFSonar 解决的是后者

如果某天文案质量稳定了,加发布器是新增一个 `Publisher` 模块的事,半小时搞定。

### 原则 5:**简单战胜花哨**

- ledger 用 JSONL,不用 SQLite
- 调度用 `time.sleep(interval)`,不用 cron / systemd-timer
- 配置用 `tomllib`(Python 3.11+ 标准库),不用 pydantic
- 没有 web UI,只有 `list` 子命令
- 没有 plugin 系统,只有一个 `Source` 抽象类 + 4 个具体实现

每加一个抽象层都要付出"未来读代码的人要多理解一层"的代价。MVP 阶段的目标是**让代码贴地飞**,等真出现需要 SQLite 的场景了再换。

## 五、模块详解

### `src/events.py` — 统一事件结构

```python
@dataclass
class Event:
    source: str          # "trending_models" | "new_models" | "daily_papers" | "watchlist"
    event_id: str        # HF 上的唯一 id (model id 或 arxiv id)
    title: str
    url: str
    summary: str
    author: str
    likes: int
    downloads: int
    created_at: str
    tags: list[str]
    extra: dict          # 信源特有的杂项
    fetched_at: str

    @property
    def dedup_key(self) -> str:
        return f"{self.source}::{self.event_id}"
```

**为什么要有这层抽象?**——4 个信源返回的数据结构完全不同(`ModelInfo` vs API JSON),如果不统一,后面的过滤/curator/writer 就要写 4 套。

`dedup_key` 是 `{source}::{event_id}` 的复合键。同一个模型 ID 可能既出现在 trending 也出现在 watchlist——这不算重复,因为关注点不同。

### `src/ledger.py` — 跨运行去重

```python
class Ledger:
    def has(self, key: str) -> bool: ...
    def filter_unseen(self, keys: Iterable[str]) -> list[str]: ...
    def record(self, key: str, *, source: str = "", note: str = ""): ...
```

启动时把 `ledger.jsonl` 的每行 JSON 读进一个 `set`,运行期所有 `has`/`filter` 都是 O(1)。`record` 既写文件又更新 set,**幂等**(重复 record 同一个 key 只写一次)。

### `src/sources/*.py` — 4 个信源适配器

每个文件实现一个 `Source` 抽象类:

```python
class Source(ABC):
    name: str
    def fetch(self, limit: int) -> list[Event]: ...
```

`fetch` 失败时**返回 `[]` 而不是 raise**——一个信源挂了不能拖累其他信源。具体实现里把异常 catch 住、log 一行、空列表返回。

`_hf_common.py` 里的 `model_info_to_event` 把 `huggingface_hub` 返回的 `ModelInfo` 对象转成统一的 `Event`。这个转换器集中在一处,以后 HF SDK 改 schema 只改这一个文件。

### `src/operator.py` — Claude CLI 包装层

定义了 `Operator` Protocol:

```python
class Operator(Protocol):
    def run(self, prompt_path: Path, *, label: str = "") -> OperatorResult: ...
```

两个具体实现:

**`ClaudeOperator`** — 真调:

```python
cmd = ["claude", "-p", f"@{prompt_path}", "--output-format", "json"]
# subprocess.run, parse stdout JSON, extract payload["result"]
```

`-p @file` 让 CLI 从文件读 prompt(便于缓存、便于回溯)。`--output-format json` 让输出变成结构化 JSON,我们只取 `result` 字段。

**`FakeOperator`** — 假调:

读 prompt 文件,根据里面的 `[ROLE: CURATOR]` / `[ROLE: WRITER]` 标记返回不同的固定结果。Curator 的假返回会从 prompt 里 grep 出前 3 个候选 id;Writer 的假返回是带"fake-llm draft"标记的 markdown。

足够让 e2e 流水线跑通,但永远不写出真正可发的内容——避免有人忘了关 `--fake-llm` 还以为质量挺好。

### `src/orchestrator.py` — 编排核心

`run_cycle` 是入口,做 6 件事:

```
1. _utc_run_id() → 生成微秒级时间戳作为 run id
2. fetch_all()    → 并行从 4 个信源抓数据
3. filter_candidates() → ledger 去重 + min_likes 阈值过滤
4. 组装 curate prompt → 调 operator → 解析 JSON 结果
5. 对每条被选中的 event,组装 writer prompt → 调 operator → 落盘 post
6. 写 run_manifest.json,返回 run_dir 路径
```

`_parse_curator_output` 是宽容的 JSON 解析器:可以接受裸 JSON、```json fenced block、prose 中嵌入的 JSON 三种形式。Claude 偶尔会"啰嗦"一下,这个解析器把那些情况都兜住。

### `src/prompts/01_curate.md` & `02_write_post.md` — Prompt 模板

不用 Jinja2 不用任何模板引擎,**就是 `text.replace("{{KEY}}", value)`**。够用。

模板里的占位符:

```
01_curate.md:
{{CYCLE_TIMESTAMP}}, {{NUM_CANDIDATES}}, {{TOP_K}}, {{CANDIDATES_BLOCK}}

02_write_post.md:
{{SOURCE}}, {{TITLE}}, {{URL}}, {{AUTHOR}}, {{LIKES}}, {{CREATED_AT}},
{{TAGS}}, {{SUMMARY}}, {{ANGLE}}
```

改 prompt 不需要改 Python。两份模板都用 `[ROLE: CURATOR]` / `[ROLE: WRITER]` 标记,`FakeOperator` 用这个标记区分要返回什么假结果。

### `main.py` — CLI 入口

`argparse`,3 个子命令(`poll` / `loop` / `list`)。loop 模式安装 SIGINT/SIGTERM handler,sleep 时按 1 秒一片切,确保 Ctrl+C 响应。

## 六、一次完整 cycle 走读

假设你在终端跑 `python main.py poll`,从那一刻开始:

```
T+0.00s   读 config.toml
T+0.01s   ClaudeOperator 实例化(检查 PATH 上有 claude)
T+0.02s   Ledger 加载 state/ledger.jsonl,假设里面有 1500 条历史 key
T+0.03s   生成 run_id = "20260508T223412_891045Z"
T+0.03s   mkdir runs/20260508T223412_891045Z/prompts/

[fetch 阶段]
T+0.20s   trending_models 拉到 20 条
T+0.50s   new_models 拉到 30 条
T+1.20s   daily_papers 拉到 15 条
T+3.50s   watchlist 串行扫 10 个 org,共拿到 28 条(10 个 org × 最多 3 条)
          总计 fetched = 93 条 events

[filter 阶段]
T+3.51s   按 source::event_id 去重(同一个 model 可能在 trending+new 都出现)→ 87 条
T+3.52s   去掉 ledger 已有的 → 41 条
T+3.53s   去掉 likes < 5 的 trending/new(papers/watchlist 豁免)→ 32 条候选

[curate 阶段]
T+3.54s   渲染 01_curate.prompt.md(把 32 条候选信息嵌入模板)
T+3.55s   写到 runs/.../prompts/01_curate.prompt.md
T+3.56s   subprocess: claude -p @01_curate.prompt.md --output-format json
T+12.4s   Claude 返回 JSON,parse 出 chosen = 5 条 + 每条的 angle

[write × 5 阶段]
T+12.5s   渲染 02_write_01.prompt.md(填入 event 信息和 angle)
T+12.5s   subprocess: claude -p @02_write_01.prompt.md ...
T+18.2s   Claude 返回 markdown body
T+18.3s   写 runs/.../posts/01__source__id.md(加 frontmatter)
T+18.3s   ledger.record(key) → state/ledger.jsonl 追加一行
... × 4 次 ...

[收尾]
T+45.0s   写 runs/.../run_manifest.json(候选列表、curator 原始输出、5 个 post 路径)
T+45.0s   stdout 打印 "run complete: runs/20260508T223412_891045Z, posts written: 5"
T+45.0s   进程退出
```

整个 cycle 大约 30~60 秒,主要时间花在 6 次 Claude CLI 调用上(1 curate + 5 write)。

## 七、配置项

`config.toml`(可被 `config.local.toml` 覆盖,后者 gitignore):

| 配置 | 默认值 | 含义 |
| --- | --- | --- |
| `[poll] trending_limit` | 20 | trending 信源每 cycle 抓多少 |
| `[poll] new_models_limit` | 30 | new 信源每 cycle 抓多少 |
| `[poll] daily_papers_limit` | 15 | papers 信源每 cycle 抓多少 |
| `[poll] watchlist_limit` | 3 | watchlist **每个 org** 抓多少(注意是 per-org) |
| `[curation] top_k` | 5 | curator 最多保留几条 |
| `[curation] min_likes` | 5 | trending/new 的赞数阈值(papers/watchlist 豁免) |
| `[watchlist] orgs` | 10 家 | 关注的 HF 组织 |
| `[claude] binary` | `"claude"` | CLI 名字 |
| `[claude] timeout` | 180 | 单次 CLI 调用超时(秒) |
| `[claude] model` | `""` | 模型覆盖,空 = 用 Claude Code 的默认 |

## 八、扩展指南

### 加一个新信源(比如 HF Spaces)

1. 在 `src/sources/` 加 `spaces.py`,实现 `Source` 接口,`fetch()` 返回 `list[Event]`
2. 在 `events.py` 加 `SOURCE_SPACES = "spaces"` 常量
3. 在 `orchestrator.build_sources()` 里加一行
4. 在 `config.toml` 的 `[poll]` 加 `spaces_limit = ...`
5. 不需要改 curator/writer/ledger/operator——它们都看的是统一的 `Event` 结构

### 加一个发布器(比如 Discord webhook)

1. 创建 `src/publishers/discord.py`,定义 `DiscordPublisher`,接 webhook URL,把 `Event + post body` 发到频道
2. 在 `orchestrator.run_cycle()` 末尾,在 `write_post_file()` 之后加一句:
   ```python
   if publisher:
       publisher.publish(evt, body)
   ```
3. 在 CLI 加 `--publish-to discord` 标志
4. 加 `[discord] webhook_url = ...` 配置

整个改动应该 30 行 Python 之内。

### 换掉 Claude

1. 在 `src/operator.py` 加一个 `OpenAIOperator` 类,实现 `Operator` Protocol
2. CLI 加 `--backbone openai` 标志
3. 完事

Curator / Writer prompts 都是纯文本,任何 LLM 都吃。

| `test_ledger.py` | 写读、幂等、`filter_unseen` |
| `test_events.py` | dedup_key、dict round-trip、未知字段容忍 |
| `test_curator_parse.py` | 裸 JSON / fenced JSON / prose 内嵌 JSON / 垃圾输入 |
| `test_orchestrator_fake.py` | FakeOperator + stub source 的 e2e + 第二轮全去重 |

故意**不测真 Claude 调用** —— 那部分依赖外部进程,容易 flaky 也烧 token。靠 `subprocess.run` 这层 Python 标准库的稳定性兜底,加上一次手动 `claude -p @file` smoke 验证 JSON shape 即可。

---

## 十、故意不做的事(以及触发线)

- ❌ **不接社交平台 API** — 等文案质量稳定 + 你想好账号策略后再做
- ❌ **没有 Web UI** — `python main.py list` + 文件浏览器够用,有 UI 反而维护成本陡增
- ❌ **没有 reviewer agent** — 只有写发布器之后才有意义(发出去前要不要再让一个 Claude 做 final check)
- ❌ **没有图片/视频生成** — 视觉模型的发布需要看图,但这是 v2 的事
- ❌ **没有多语言** — Writer prompt 写死英文,加中文 prompt 就是再加一个模板文件
- ❌ **没有 SQLite/Postgres** — JSONL 在百万级以下都够;真到了就换

每一条都不是"做不到",而是**当下不值得做**。理由统一:**先把 v1 跑稳定 → 看哪条限制真的影响了你 → 再去做那一条**。

---

## 十一、一句话总结

> HFSonar 是个 **"用 Python 把 Claude Code 抱在一个 polling loop 里" 的最小可用 agent**。
> 它把"刷 HuggingFace + 决定值不值得发 + 写第一版稿"这件每天耗你半小时的活儿,自动化成 5 个磁盘上的 markdown 草稿。
> 它故意不替你按"发送键"——因为按不按那个键,本来就该是你来决定的。
